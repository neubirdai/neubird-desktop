    Documentation Development Guide

    This directory contains the source for the NeuBird Desktop documentation site, built with MkDocs Material.

    Quick Start

    # Install Python dependencies (once)
pip install mkdocs-material

# Serve docs locally at http://localhost:8000
make docs

    How It Works

    The docs system has three layers:

    1. Content Generation (make docs-gen)

    Runs automatically as part of make docs. Generates:

    • docs/index.md — produced from README.md with a transform that swaps the GitHub asciinema badge for an inline interactive player
    • docs/configuration/flags.md — auto-generated from Go source code (internal/docmeta/)
    • docs/configuration/env-vars.md — auto-generated from Go source code

    You never edit docs/index.md directly — edit README.md instead.

    2. Demo Recordings (make record-demos)

    Run manually when demo scenarios or TUI rendering code changes. Requires external tools:

    # Install recording dependencies (once)
brew install asciinema charmbracelet/tap/freeze
# tmux is usually already installed

    This builds the neubird binary, then for each demos/*.yaml scenario:

    1. Runs asciinema rec → produces docs/assets/casts/<scenario>.cast
    2. Runs tmux + freeze → produces docs/assets/screenshots/<scenario>-<marker>.svg
    3. Writes .hash sidecars for staleness checking

    The generated .cast and .svg files are committed to git — CI doesn't re-record them.

    3. Site Build & Deploy

    • Locally: make docs (runs docs-gen + mkdocs serve)
    • CI: .github/workflows/docs.yml runs on push to main, builds the site, and deploys to the neubird-desktop public repo via GitHub Pages

    Directory Structure

    docs/
  index.md                  ← generated from README.md (don't edit)
  assets/
    casts/*.cast            ← asciinema recordings (generated by record-demos)
    screenshots/*.svg       ← terminal screenshots (generated by record-demos)
    asciinema-player.*      ← vendored player v3.15.1
    logo-color.svg          ← full NeuBird wordmark
    icon.svg                ← bird icon (navbar/favicon)
    neubird.css             ← brand colors
  features/                 ← feature documentation pages
  getting-started/          ← installation, quickstart
  usage/                    ← TUI, slash commands, shortcuts, personas
  configuration/            ← auto-generated flags and env vars
  superpowers/              ← internal specs/plans (excluded from build)

demos/
  *.yaml                    ← demo scenario definitions (embedded in binary)

tools/
  gendocs/                  ← generates flags.md and env-vars.md from Go source
  record-demos/             ← orchestrates asciinema + freeze recordings
  check-demos/              ← validates asset freshness via hash sidecars

    Adding a New Feature Page

    1. Create docs/features/my-feature.md
    2. Add it to the nav: section in mkdocs.yml
    3. If it needs a recording, create demos/my-feature.yaml and run make record-demos
    4. Embed the recording using direct HTML (not the MkDocs plugin — it doesn't handle v3 casts):
    <link rel="stylesheet" type="text/css" href="../../assets/asciinema-player.css" />
<div id="my-feature-player"></div>
<script src="../../assets/asciinema-player.min.js"></script>
<script>AsciinemaPlayer.create("../../assets/casts/my-feature.cast",
  document.getElementById("my-feature-player"),
  {cols: 120, rows: 35, autoPlay: true, fit: "width", terminalFontSize: "small"});</script>

    Demo Scenario Format

    See demos/health-check.yaml for the full format. Key fields:

    name: My Feature Demo
description: What this demo shows
terminal:
  width: 120
  height: 35
  idle_limit: 2s
  linger: 3s
question: "/health"        # auto-typed into the TUI input

markers:                   # screenshot capture points
  - id: final-state
    after_event: -1        # -1 = after done event
    wait: 500ms

events:
  - kind: text             # text | tool_start | tool_result | progress | sources | done
    delay: 500ms
    text: "Investigation text..."

    Makefile Targets

    Target What it does When to run
    make docs Generate content + serve locally Day-to-day editing
    make docs-gen Generate index.md + config pages only CI, or to check generation
    make record-demos Record all demo scenarios After changing demos/*.yaml or TUI code
    make check-demos Verify recordings are up-to-date Before committing (optional)
