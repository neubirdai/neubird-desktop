    SSO Login Implementation

    This document describes the SSO (Single Sign-On) login feature added to Neubird Falcon CLI.

    Overview

    Falcon now supports OAuth 2.0 Authorization Code flow with PKCE (Proof Key for Code Exchange) for secure browser-based authentication. This allows users to log in using their organization's SSO provider (Okta, Azure AD, Google, etc.) instead of entering username/password directly in the CLI.

    Architecture

    Components

    1. OAuth Client (internal/oauth/client.go)
    2. Implements OAuth 2.0 Authorization Code flow with PKCE
    3. Generates cryptographically secure code verifier and challenge
    4. Starts local HTTP callback server on ports 8080-8099
    5. Opens user's browser to SSO provider

    6. Exchanges authorization code for access token

    7. Browser Opener (internal/oauth/browser.go)

    8. Cross-platform browser launching (macOS, Linux, Windows)

    9. Falls back to manual URL copy if browser can't be opened

    10. Login Command (internal/login/login.go)

    11. CLI command: neubird login <url> --sso
    12. Supports both SSO and legacy username/password flows

    13. Auto-detects SSO when no credentials provided

    14. TUI Login (internal/tui/login.go, internal/tui/login_sso.go)

    15. Interactive terminal UI with SSO support
    16. Press S key to trigger SSO login

    17. Shows authentication status during flow

    18. Credentials Storage (internal/config/credentials.go)

    19. Extended to store refresh_token and expires_at
    20. Maintains backward compatibility with password-based auth

    Usage

    Command Line

    # SSO login (opens browser)
neubird login https://myenv.app.neubird.ai --sso

# Auto-detect SSO (no username/password provided)
neubird login https://myenv.app.neubird.ai

# Legacy password login still works
neubird login https://myenv.app.neubird.ai -u user@company.com -p password

    Interactive TUI

    When the TUI login screen appears:

    1. Enter the NeuBird URL
    2. Press S to trigger SSO login
    3. Browser opens automatically
    4. Complete authentication in browser
    5. Return to terminal - login complete!

    Configuration

    Client ID

    The OAuth client ID can be configured via environment variable:

    export OAUTH_CLIENT_ID="my-custom-client-id"

    Default: falcon-cli

    Backend Endpoints

    The implementation expects these OAuth endpoints on the Raven backend:

    • GET /v1/auth/authorize - Authorization endpoint
    • POST /v1/auth/token - Token exchange endpoint

    Security Features

    ✅ PKCE - Prevents authorization code interception attacks
    ✅ Localhost redirect - Tokens never leave the local machine
    ✅ Secure storage - Credentials saved with 0600 permissions
    ✅ Token refresh - Supports refresh tokens for long-lived sessions
    ✅ Timeout protection - 5-minute timeout on callback server
    ✅ State parameter - CSRF protection (can be added if needed)

    Token Refresh

    The OAuth client includes a RefreshToken() method for obtaining new access tokens:

    client := oauth.NewClient(ravenURL, "")
newToken, err := client.RefreshToken(ctx, refreshToken)

    This can be integrated into the main application to automatically refresh expired tokens.

    Testing

    Manual Testing

    1. Happy path:

    bash neubird login https://myenv.app.neubird.ai --sso

    • Browser should open
    • Complete SSO login
    • Return to terminal

    • Check ~/.config/neubird/neubird.json for saved tokens

    • Error handling:

    • Test with invalid URL
    • Test with network disconnected
    • Test with browser that won't open

    • Test timeout (wait 5+ minutes)

    • TUI mode:

    • Run neubird without credentials
    • Press S for SSO
    • Verify browser opens and flow completes

    Integration Testing

    The OAuth client can be tested against a mock OAuth server:

    // Start mock server
server := httptest.NewServer(http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
    if r.URL.Path == "/v1/auth/authorize" {
        // Redirect to callback with code
        code := "test-auth-code"
        redirectURI := r.URL.Query().Get("redirect_uri")
        http.Redirect(w, r, redirectURI+"?code="+code, http.StatusFound)
    } else if r.URL.Path == "/v1/auth/token" {
        // Return token
        json.NewEncoder(w).Encode(map[string]string{
            "access_token": "test-access-token",
            "refresh_token": "test-refresh-token",
            "expires_in": "3600",
        })
    }
}))
defer server.Close()

// Test OAuth flow
client := oauth.NewClient(server.URL, "test-client")
token, err := client.StartAuthFlow(context.Background())

    Troubleshooting

    Browser doesn't open

    If the browser doesn't open automatically, the URL is printed to the terminal. Copy and paste it manually.

    Port already in use

    The callback server tries ports 8080-8099. If all are in use, you'll see an error. Close other applications using these ports.

    Token expired

    If you see authentication errors after some time, the access token may have expired. The refresh token flow can be implemented to automatically renew tokens.

    Firewall blocks localhost

    Some corporate firewalls block localhost connections. In this case, the device code flow (not yet implemented) would be a better option.

    Future Enhancements

    1. Device Code Flow - For headless/SSH environments where browser can't be opened
    2. State Parameter - Additional CSRF protection
    3. Automatic Token Refresh - Refresh tokens before they expire
    4. Multiple SSO Providers - Support different providers per environment
    5. Token Revocation - Logout endpoint to revoke tokens

    Files Changed

    • internal/oauth/client.go - OAuth client implementation
    • internal/oauth/browser.go - Browser opening utility
    • internal/config/credentials.go - Extended credentials structure
    • internal/login/login.go - Added SSO login command
    • internal/login/oauth.go - OAuth client wrapper
    • internal/tui/login.go - Added SSO trigger in TUI
    • internal/tui/login_sso.go - TUI SSO login implementation

    Dependencies

    No new external dependencies required! All implementation uses Go standard library:

    • crypto/rand - Secure random generation
    • crypto/sha256 - PKCE challenge hashing
    • encoding/base64 - Base64URL encoding
    • net/http - HTTP client and callback server
    • os/exec - Browser launching
